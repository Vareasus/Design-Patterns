SOLID

S

Single Responsibility Principle

"A class should have one, and only one, reason to change." — Robert C. Martin

🍕 The Pizza Restaurant Analogy

In a pizzeria, the chef makes pizzas, the cashier handles payments, and the delivery driver delivers orders. If one person did all three, when the menu changes, the tax laws change, OR delivery routes change — the same person needs retraining. Each role change affects the same person. That's an SRP violation. Each role should be separate.

❌ BAD — Three responsibilities in one class
// This class has THREE reasons to change
public class Invoice
{
    public decimal Amount { get; set; }
    public string Customer { get; set; }

    // Reason 1: Calculation logic
    public decimal CalculateTotal(decimal tax)
        => Amount + (Amount * tax);

    // Reason 2: Display/Print format
    public string PrintInvoice()
        => $"Invoice: {Customer} — ${Amount}";

    // Reason 3: Database persistence
    public void SaveToDatabase()
    {
        // Direct SQL connection inside model 💥
        var conn = new SqlConnection("...");
    }
}
✅ GOOD — One responsibility per class
// Data only
public class Invoice
{
    public decimal Amount { get; set; }
    public string Customer { get; set; }
}

// Calculation only
public class InvoiceCalculator
{
    public decimal CalculateTotal(Invoice inv, decimal tax)
        => inv.Amount + (inv.Amount * tax);
}

// Printing only
public class InvoicePrinter { /* ... */ }

// Persistence only
public class InvoiceRepository { /* ... */ }
🎯

One Reason to Change

Not "one thing to do." A class should be responsible to ONE actor/stakeholder.

🔍

Red Flag: "And"

If a class name needs "and" to describe it — it does too much.

🧪

Easy to Test

Focused classes are trivially testable. God classes need the whole universe to test.

🗣️ Interview Answer

"SRP states that a class should have only one reason to change, meaning it should be responsible to only one actor. For example, an Invoice class that calculates totals, prints reports, AND saves to database violates SRP. Instead, I separate those into InvoiceCalculator, InvoicePrinter, and InvoiceRepository — each with a single, focused responsibility. This makes the code testable, maintainable, and reduces merge conflicts."

O

Open/Closed Principle

"Software entities should be open for extension, but closed for modification." — Bertrand Meyer

🔌 The Power Outlet Analogy

A wall power outlet is closed for modification (you don't rewire it for each new device) but open for extension (you can plug in ANY device). The outlet doesn't know or care if it's powering a lamp, laptop, or guitar amp — it just provides power through a standard interface.

❌ BAD — Growing switch statement
public class AreaCalculator
{
    public double Calculate(object shape)
    {
        if (shape is Circle c)
            return Math.PI * c.Radius * c.Radius;
        else if (shape is Rectangle r)
            return r.Width * r.Height;
        else if (shape is Triangle t)
            return 0.5 * t.Base * t.Height;
        // Pentagon? Hexagon? Star?
        // This grows FOREVER 💥
        throw new ArgumentException("Unknown");
    }
}
✅ GOOD — Extend without modifying
public interface IShape
{
    double CalculateArea();
}

public class Circle : IShape
{
    public double CalculateArea()
        => Math.PI * Radius * Radius;
}

// NEW shape? Just add a class!
public class Pentagon : IShape
{
    public double CalculateArea()
        => /* formula */;
}

// Calculator NEVER changes ✅
public class AreaCalculator
{
    public double Total(IEnumerable<IShape> s)
        => s.Sum(x => x.CalculateArea());
}
🔧

Extend, Don't Modify

Add new classes instead of editing existing ones. Working code stays untouched.

🔌

Interfaces = Extension Points

Define an interface, then implement it for each new variant.

⚠️

Switch = Code Smell

A growing if-else/switch on types usually means you need polymorphism.

🗣️ Interview Answer

"OCP means we should be able to extend a system's behavior without modifying existing source code. I use abstractions like interfaces as extension points. For example, instead of a giant if-else chain for shape area calculations, each shape implements an IShape interface with CalculateArea(). Adding a Pentagon means creating a new class — the calculator never changes. This reduces risk and makes the system naturally extensible."

L

Liskov Substitution Principle

"Objects of a superclass should be replaceable with objects of a subclass without affecting the correctness of the program." — Barbara Liskov, 1987

🦆 The Duck Test Gone Wrong

"If it looks like a duck and quacks like a duck..." — but what about a RubberDuck? It looks like a duck but can't fly, can't quack (it squeaks), and doesn't swim (it floats). Code expecting a Duck will break when it gets a RubberDuck. Subtypes must honor the full contract of their parent.

❌ BAD — Square breaks Rectangle's contract
public class Rectangle
{
    public virtual int Width { get; set; }
    public virtual int Height { get; set; }
    public int Area() => Width * Height;
}

public class Square : Rectangle
{
    public override int Width {
        set { base.Width = value;
               base.Height = value; } // 💥
    }
}

// rect.Width = 5; rect.Height = 10;
// Expected: 50. Got: 100! 💥
✅ GOOD — Shared interface, no broken contracts
public interface IShape
{
    int CalculateArea();
}

public class Rectangle : IShape
{
    public int Width { get; set; }
    public int Height { get; set; }
    public int CalculateArea()
        => Width * Height;
}

public class Square : IShape
{
    public int Side { get; set; }
    public int CalculateArea()
        => Side * Side;
}
// Both work through IShape ✅
📝

Honor the Contract

Subtypes must fulfill ALL promises of the parent — behavior, not just signatures.

🚫

No Surprise Exceptions

If the parent doesn't throw, the child shouldn't throw. NotImplementedException = LSP violation.

🔄

The Swap Test

Can you replace the parent with the child and have ALL tests still pass?

🗣️ Interview Answer

"LSP says that a subclass must be usable wherever its parent is used without breaking the program. The classic example is Rectangle vs Square — mathematically a square is a rectangle, but in code, Square extending Rectangle causes bugs because setting width independently violates the square's invariant. The fix: use a shared interface like IShape instead of forced inheritance. Key test: can I swap in the subclass and still pass all existing tests?"

I

Interface Segregation Principle

"No client should be forced to depend on methods it does not use." — Robert C. Martin

🍽️ The Restaurant Menu Analogy

A restaurant that hands you a 200-page menu covering breakfast, lunch, dinner, desserts, catering, and party packages. You came for a sandwich at noon. You don't need the party planner section. ISP says: give the lunch customer a lunch menu. Small. Focused. Relevant.

❌ BAD — Fat interface forces robot to eat and sleep
public interface IWorker
{
    void Work();
    void Eat();
    void Sleep();
}

public class RobotWorker : IWorker
{
    public void Work() { /* ✅ Sure */ }

    public void Eat()
        => throw new NotImplementedException();
        // 💥 Robots don't eat!

    public void Sleep()
        => throw new NotImplementedException();
        // 💥 Robots don't sleep!
}
✅ GOOD — Small interfaces, pick what you need
public interface IWorkable
{  void Work(); }

public interface IFeedable
{  void Eat(); }

public interface ISleepable
{  void Sleep(); }

// Human does all three
public class Human :
    IWorkable, IFeedable, ISleepable
{ /* implements all */ }

// Robot ONLY works — nothing forced!
public class Robot : IWorkable
{
    public void Work() { /* 24/7! */ }
}
✂️

Split Fat Interfaces

Many small interfaces beat one giant interface. Always.

🎯

Client-Driven Design

Design interfaces based on what CLIENTS need, not what IMPLEMENTORS have.

🚩

NotImplementedException = Red Flag

If you're throwing it, the interface is too fat for this class.

🗣️ Interview Answer

"ISP states that clients shouldn't be forced to depend on interfaces they don't use. Instead of one large IWorker interface with Work, Eat, Sleep, and WriteReport, I split it into IWorkable, IFeedable, ISleepable, and IReportable. A RobotWorker only implements IWorkable — not forced to throw NotImplementedException for Eat. This makes code more maintainable and prevents dead weight."

D

Dependency Inversion Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions." — Robert C. Martin

🔌 The Phone Charger Analogy

Your phone doesn't have a wire permanently soldered to a wall outlet. Instead, it depends on a USB-C port (abstraction). You can use ANY charger — wall, car, power bank — as long as it speaks USB-C. The phone doesn't know or care WHERE the power comes from. That's dependency inversion.

❌ BAD — Directly coupled to MySQL
public class UserService
{
    // 💥 Tightly coupled to MySQL!
    private readonly MySqlDatabase _db
        = new MySqlDatabase();

    public void CreateUser(string name)
    {
        _db.Save(name);
        // Want PostgreSQL? Change THIS class.
        // Want to unit test? Need real MySQL.
        // 💥 Rigid. Untestable. Fragile.
    }
}
✅ GOOD — Depends on abstraction
public interface IDatabase
{  void Save(string data); }

public class MySql : IDatabase { /* ... */ }
public class Postgres : IDatabase { /* ... */ }
public class MongoDB : IDatabase { /* ... */ }

public class UserService
{
    private readonly IDatabase _db;

    // Inject ANY database!
    public UserService(IDatabase db)
    { _db = db; }

    public void CreateUser(string name)
    { _db.Save(name); }
}
// ✅ Swap databases in config, not code
⬆️

Invert the Direction

High-level defines interfaces. Low-level implements them. Dependencies flow toward abstractions.

💉

DIP ≠ DI

DIP is the PRINCIPLE (depend on abstractions). Dependency Injection is a TECHNIQUE to achieve it.

🧪

Testability Unlocked

Inject mocks for tests. No real databases, emails, or APIs needed.

🗣️ Interview Answer

"DIP says high-level business logic shouldn't directly depend on low-level infrastructure. Both depend on abstractions. Instead of UserService creating a MySqlDatabase instance, I define an IDatabase interface. UserService receives IDatabase through its constructor — it doesn't know if it's MySQL, PostgreSQL, or a mock for testing. This makes the system flexible, testable, and easy to evolve without touching business logic."

🏗️ Design Patterns

Gang of Four Patterns

Proven, reusable solutions to common problems. These patterns are implementations of SOLID principles.

🎯

Strategy Pattern

"Define a family of algorithms, encapsulate each one, and make them interchangeable." — Gang of Four

🎮 The Game Controller Analogy

A game console works with any controller — wired, wireless, fight stick, racing wheel. The console doesn't care which one you use. It knows the interface: buttons, joystick, triggers. Plug in any controller and play. That's Strategy.

❌ BAD — Hardcoded algorithms
public List<int> Sort(List<int> data, string algo)
{
    switch (algo)
    {
        case "bubble": /* bubble sort */
        case "quick":  /* quick sort */
        case "merge":  /* merge sort */
        // 💥 New algorithm = modify this method
    }
}
✅ GOOD — Swappable strategies
public interface ISortStrategy
{  List<int> Sort(List<int> data); }

public class BubbleSort : ISortStrategy { /*...*/ }
public class QuickSort : ISortStrategy { /*...*/ }

public class SortingService
{
    private ISortStrategy _strategy;
    public void SetStrategy(ISortStrategy s)
        => _strategy = s;
    public List<int> Sort(List<int> data)
        => _strategy.Sort(data);
}
// ✅ Swap at runtime. Zero changes.
🔄

Runtime Swapping

Change algorithms on the fly without modifying client code.

🔌

OCP + DIP

New strategies = new classes. Context depends on interface.

📦

Use When

Multiple algorithms for the same task: sorting, pricing, validation, compression.

📡

Observer Pattern

"Define a one-to-many dependency so that when one object changes state, all dependents are notified." — Gang of Four

📰 The Newsletter Analogy

You subscribe to a newsletter. When a new article is published, everyone subscribed gets notified. You don't keep checking the website. And you can unsubscribe anytime. The publisher doesn't know who you are — just that you're on the list.

❌ BAD — Hardcoded notifications
public void NotifyRestock(string product)
{
    // 💥 Store knows about EVERY channel
    SendEmail(product);
    SendSMS(product);
    SendSlack(product);
    SendPush(product);
    // Want Telegram? Modify THIS method.
}
✅ GOOD — Subscribe/unsubscribe
public interface IObserver
{  void OnProductAvailable(ProductEvent e); }

public class Store // Subject
{
    List<IObserver> _observers = new();
    public void Subscribe(IObserver o)
        => _observers.Add(o);
    public void Notify(ProductEvent e)
        => _observers.ForEach(o => o.OnProductAvailable(e));
}

public class EmailNotifier : IObserver { /*...*/ }
public class SlackNotifier : IObserver { /*...*/ }
// ✅ Add/remove observers freely.
📢

One-to-Many

One subject, many observers. State changes propagate automatically.

🔓

Loose Coupling

Subject doesn't know concrete observers. Only the interface.

📦

Use When

Event systems, notifications, real-time feeds, pub/sub messaging.

🏭

Factory Pattern

"Define an interface for creating an object, but let subclasses decide which class to instantiate." — Gang of Four

🍔 The Restaurant Kitchen Analogy

You tell the waiter "I want a burger." The kitchen (factory) decides how to make it — ingredients, cooking method, which chef. You don't go into the kitchen yourself. You just get a burger that satisfies the IFood interface.

❌ BAD — Direct new with switch
public object CreateDoc(string type)
{
    switch (type)
    {
        case "pdf": return new PdfDoc();
        case "word": return new WordDoc();
        case "excel": return new ExcelDoc();
        // 💥 PowerPoint? Modify here.
    }
}
✅ GOOD — Factory creates through abstraction
public interface IDocument
{
    void Generate();
    void Save(string path);
}

public class PdfDoc : IDocument { /*...*/ }
public class WordDoc : IDocument { /*...*/ }

public static IDocument Create(string type)
    => _creators[type]();

// Register new types without modifying!
Register("md", () => new MarkdownDoc());
// ✅ Client only knows IDocument
🔨

Delegate Creation

Client asks for an object. Factory decides which concrete class.

🔌

OCP + DIP

New types = register them. Client code stays unchanged.

📦

Use When

Complex object creation, multiple product families, plugin systems.

🔒

Singleton Pattern

"Ensure a class has only one instance and provide a global point of access to it." — Gang of Four

🏛️ The President Analogy

A country has only one president at a time. Everyone refers to the same person. You don't "create a new president" — you access the existing one. Singleton ensures only one instance exists application-wide.

❌ BAD — Not thread-safe
public class Database
{
    private static Database _instance;
    private Database() { } // Private constructor

    public static Database Instance
    {
        get
        {
            // 💥 Race condition!
            // Thread A: checks null → true
            // Thread B: checks null → true (before A creates)
            // Both threads create separate instances!
            if (_instance == null)
                _instance = new Database();
            return _instance;
        }
    }

    public void Query(string sql)
        => Console.WriteLine($"Executing: {sql}");
}

// 💥 Problems:
// 1. Not thread-safe (race condition)
// 2. Can't mock for unit tests
// 3. Hidden dependency — hard to track
// 4. Global state — can cause side effects
✅ GOOD — 3 Safe Approaches
// ── Approach 1: Lazy<T> (Thread-safe) ──
public sealed class AppConfig
{
    private static readonly Lazy<AppConfig> _inst
        = new(() => new AppConfig());

    public static AppConfig Instance => _inst.Value;
    private AppConfig() { } // Private ctor

    public string AppName { get; } = "MyApp";
    public string Env { get; } = "Production";
}

// ── Approach 2: Static (Eager Loading) ──
public sealed class Logger
{
    private static readonly Logger _inst = new();
    public static Logger Instance => _inst;
    private Logger() { }
}

// ── Approach 3: DI Container (BEST) ──
// builder.Services
//   .AddSingleton<ILogService, LogService>();
// ✅ Thread-safe + Testable + Mockable
🔒

Thread Safety

Lazy<T> guarantees only ONE instance, even with 100 threads hitting it simultaneously.

⚠️

Use Sparingly

Most controversial pattern. Can become a God Object. Creates hidden dependencies. Hurts testability.

✅

Good For

Configuration, logging, connection pools, caches — shared resources that are expensive to create.

❌

Bad For

Business logic, services, repositories — anything that should be mockable in unit tests.

💉

Modern Approach

Use DI container's AddSingleton<T>() — same single instance, but injectable and testable.

🔑

Key Elements

Private constructor + static instance + sealed class + thread-safe access.

🗣️ Interview Answer

"Singleton ensures only one instance of a class exists application-wide. I use it for shared resources like configuration or logging. In C#, the safest implementation uses Lazy<T> which guarantees thread safety. However, in modern apps I prefer registering the class as a singleton in the DI container using AddSingleton<T>() — this gives me the same single-instance behavior but with full testability since I can inject mocks. The classic Singleton creates hidden dependencies and global state, so I treat it as a last resort, not a go-to."

🎨

Decorator Pattern

"Attach additional responsibilities to an object dynamically. A flexible alternative to subclassing." — Gang of Four

☕ The Coffee Shop Analogy

Start with basic coffee ($3). Add milk (+$0.50). Add whipped cream (+$0.70). Add caramel (+$0.60). Each addition wraps the previous. You can combine them any way you want — no class explosion.

❌ BAD — Class explosion
// Need a class for EVERY combination 💥
class Coffee { }
class CoffeeWithMilk { }
class CoffeeWithMilkAndSugar { }
class CoffeeWithMilkSugarWhip { }
class CoffeeWithMilkSugarWhipCaramel { }
// CoffeeWithSoyVanillaExtraShot...???
// 💥 50+ classes needed!
✅ GOOD — Stackable wrappers
public interface IBeverage
{  string Desc { get; }  decimal Cost { get; } }

public class Coffee : IBeverage { /* $3 */ }

public class MilkDecorator : IBeverage
{
    IBeverage _inner;
    public decimal Cost => _inner.Cost + 0.50m;
}

// Stack freely!
IBeverage order = new CaramelDecorator(
  new WhipDecorator(
    new MilkDecorator(new Coffee())));
// ✅ Infinite combos. No explosion.
🧅

Layer by Layer

Each decorator adds ONE feature. Stack as many as you want.

🔌

OCP + SRP

Add behavior without modifying originals. Each decorator = one concern.

📦

Use When

Configurable features, middleware pipelines, stream wrappers.
